# Spec 31 — Typed Per-Collection Queries

## Overview

The CLI generates typed Convex query and mutation files for every collection at
`convex/vex/collections/{slug}.ts`. Each file exports `get`, `list`, `create`,
`update`, `remove`, and `search` (if a search index is defined). Media
collections get `get`, `list`, and `remove` only (uploads go through file
storage, not `createDocument`). All functions include RBAC permission checks via
`hasPermission()` — if no access config is set, they pass silently. The CLI also
generates a barrel `index.ts` and deletes stale files when collections are
removed.

## Design Decisions

- **`get` uses `vexQuery`** — adds the `_vexDrafts` arg and `ctx.drafts` mode to
  every collection, regardless of whether versions are enabled. Consistent API,
  no conditional generation needed.
- **`list` and `search` use `query` from `_generated/server`** — properly typed
  with the app's DataModel, no `_vexDrafts` arg needed.
- **`create`, `update`, `remove` use `mutation` from `_generated/server`** —
  same reason.
- **Permission checks always generated** — `hasPermission()` returns `true`
  when `access` is `undefined`, so they silently pass with no access config.
- **Auth helper imported from `../../auth`** — user creates `convex/vex/auth.ts`
  once. Documented in Step 5. Not generated by the CLI.
- **`tableName` used for Convex IDs** — `v.id(tableName)` not `v.id(slug)` when
  they differ.
- **First `searchIndex` used for `search`** — collections with multiple search
  indexes: generated `search` uses `searchIndexes[0]`.
- **Convex dir derived from `schema.outputPath`** — if outputPath is
  `/convex/vex.schema.ts`, convex dir is `convex/`. No new config field needed.
- **Stale file cleanup** — CLI detects the `// ⚠️ AUTO-GENERATED BY VEX CMS`
  header and deletes files that are no longer in the current collection set.
- **Model functions exported from `@vexcms/core`** — `getDocument`,
  `listDocuments`, etc. are already in
  `packages/core/src/convex/model/collections.ts` but not yet exported from the
  package. Step 1 adds these exports.

## Out of Scope

- Globals codegen (Spec 30 scope)
- `count` query in generated files
- `bulkDelete` in generated files (admin panel only)
- Versioning-specific queries (saveDraft, publish, restoreVersion) — Spec 07
- Index-filtered list variants — user writes custom Convex functions for that

## Implementation Order

1. **Export model functions** from `@vexcms/core` — after this step `build`
   works and generated files can import from `@vexcms/core`.
2. **`generateCollectionQueries`** in core — pure function, string output,
   fully unit-testable without file system.
3. **`generateCollectionFiles`** in CLI — write/cleanup logic; test with a temp
   directory.
4. **Wire into `generateAndWrite`** — call `generateCollectionFiles()` after
   schema generation. After this step running `vex dev` in the test app writes
   the collection files.
5. **Auth setup + test-app integration** — document `convex/vex/auth.ts`,
   verify test-app compiles.

### The `convex/vex/auth.ts` contract

Every generated collection file imports `{ getUser }` from `../../auth` (i.e.
`convex/vex/auth.ts`). This file is **not generated by the CLI** — the user
creates it once and it stays in version control. It must export a `getUser`
function with this signature:

```typescript
import type { GenericQueryCtx } from "convex/server";

export type UserAuth = {
  user: Record<string, unknown>;
  roles: string[];
};

/**
 * Returns the authenticated user and their roles, or null if unauthenticated.
 * Called by all generated collection query and mutation files.
 */
export async function getUser(
  ctx: GenericQueryCtx<any>,
): Promise<UserAuth | null> {
  // Implementation depends on your auth setup.
  // Example using Better Auth with the users collection:
  const identity = await ctx.auth.getUserIdentity();
  if (!identity?.email) return null;

  const user = await ctx.db
    .query("user") // use your users table slug
    .withIndex("by_email", (q) => q.eq("email", identity.email!))
    .first();

  if (!user) return null;

  return {
    user: user as Record<string, unknown>,
    roles: (user.role as string[]) ?? [],
  };
}
```
